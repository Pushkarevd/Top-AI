# КИМ-3.2. Лабораторные работы «Развертывание распределенного хранилища» (занятие 3.3) и «Формирование детального слоя данных (DDS) с использованием Apache Spark: очистка, преобразование, объединение и загрузка данных в целевое хранилище» (занятие 3.4)

## Назначение оценочного средства
Комплект контрольно-измерительных материалов предназначен для оценивания уровня сформированности компетенций **BD-4.1** и **PL-1.3** у обучающихся по дисциплине «Технологии хранения и обработки больших данных». Оценочное средство охватывает сквозной практический сценарий построения распределенного хранилища данных в архитектуре **Data Lakehouse**: от загрузки «сырых» данных в HDFS и их первичной очистки (занятие 3.3) до формирования детального слоя данных (DDS) с применением сложных преобразований, объединения источников и загрузки в табличное хранилище Apache Iceberg (занятие 3.4).

КИМ позволяет проверить способность обучающегося:
- организовывать централизованное хранилище данных (Data Lake) и их распределённое хранение на базе экосистемы Apache Hadoop;
- применять фреймворк Apache Spark (PySpark) для параллельной обработки больших данных;
- проектировать архитектуру хранилища (партиционирование, bucketing, структура HDFS-директорий);
- выполнять очистку, преобразование, объединение данных, проверку качества (Data Quality);
- загружать результаты в целевое хранилище Apache Iceberg с подготовкой к потоковой обработке.

### 4.2. Этапы выполнения занятия 3.3 «Развертывание распределенного хранилища»

1. Подключиться к кластеру Hadoop/Spark через SSH, проверить доступность сервисов HDFS и YARN.
2. Спланировать архитектуру распределенного хранилища: выбрать стратегию организации директорий в HDFS, схему партиционирования, формат хранения.
3. Самостоятельно скачать «сырой» датасет US Used Cars Dataset с платформы Kaggle и загрузить его в HDFS с использованием консольного клиента `hdfs dfs`.
4. Настроить сессию Apache Spark с интеграцией Apache Iceberg (подключить необходимые пакеты, указать warehouse-путь).
5. Создать базу данных и «сырую» таблицу (Raw Layer) в формате Apache Iceberg.
6. Выполнить первичную очистку данных: приведение типов, обработка пропущенных значений, удаление дубликатов, фильтрация выбросов (ограничения: `daysonmarket ≤ 500`, `horsepower < 400`, `mileage ≤ 200000`, `price ≤ 60000`, `maximum_seating < 20`).
7. Провести разведочный анализ данных (EDA) с использованием PySpark DataFrame API.
8. Сформировать производные признаки: `is_any_cert` (объединение бинарных признаков сертификации), `age` (возраст автомобиля).
9. Сохранить очищенные данные в целевую Iceberg-таблицу с применением выбранной стратегии партиционирования.
10. Проверить корректность развернутого хранилища: целостность данных, структуру партиций, доступ через Spark SQL.
11. Подготовить отчёт о проделанной работе.

> **Примечание:** полученная в результате выполнения работы Iceberg-таблица станет основой для выполнения следующей лабораторной работы (занятие 3.4 — формирование детального слоя данных DDS).

### 4.3. Этапы выполнения занятия 3.4 «Формирование детального слоя данных (DDS)»

1. Подключиться к кластеру Hadoop/Spark, настроить сессию Spark с интеграцией Apache Iceberg.
2. Загрузить сырые данные из распределённого хранилища (HDFS) в Spark DataFrame из нескольких источников (основной датасет + синтетический справочник согласно варианту).
3. Выполнить базовую очистку данных: обработка пропущенных значений, приведение типов, удаление дубликатов, фильтрация аномалий.
4. Выполнить преобразование данных: фильтрация, агрегация, применение оконных функций, создание производных признаков.
5. Выполнить объединение данных из нескольких источников с использованием различных типов JOIN.
6. Провести проверки качества данных (Data Quality): полнота, уникальность, согласованность.
7. Загрузить обработанные данные в целевое хранилище (таблица Apache Iceberg в HDFS). 
8. Подготовить отчёт о проделанной работе.

### 4.4. Варианты заданий (по номеру студента в группе mod 4)

1. **Индивидуальное выполнение:** Работа выполняется каждым обучающимся самостоятельно.

История коммитов должна отражать поэтапное выполнение работы, а не единичную загрузку финального результата перед сдачей.
