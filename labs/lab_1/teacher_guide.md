### Вводная лекция (тезисы)

**OLTP системы:**
- Online Transaction Processing
- Нормализованная структура (3NF)
- Частые INSERT/UPDATE/DELETE
- Короткие транзакции
- Примеры: банки, интернет-магазины, CRM

**OLAP системы:**
- Online Analytical Processing
- Денормализованная структура (схема "звезда")
- Преимущественно SELECT
- Длинные транзакции, большие данные
- Примеры: хранилища данных, BI-системы

**Планы выполнения:**
- Seq Scan vs Index Scan
- Nested Loop, Hash Join, Merge Join
- Cost, Actual Time, Rows, Width

## 6. ТИПИЧНЫЕ ОШИБКИ СТУДЕНТОВ

### Ошибки в проектировании

**OLTP:**
- Отсутствие внешних ключей
- Пропуск индексов на внешних ключах
- Нарушение нормальных форм

**OLAP:**
- Нормализация вместо денормализации
- Отсутствие таблицы фактов
- Неправильный выбор гранулярности

### Ошибки в SQL

```sql
-- Ошибка: Cartesian product
SELECT * FROM orders, order_items;

-- Правильно:
SELECT * FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id;

-- Ошибка: GROUP BY без агрегации
SELECT user_id, COUNT(*) FROM orders GROUP BY user_id, order_date;

-- Правильно:
SELECT user_id, COUNT(*) FROM orders GROUP BY user_id;
```

### Ошибки в анализе планов

- Непонимание разницы между cost и actual time
- Игнорирование количества строк (Rows)
- Неверная интерпретация типов соединений

## 8. АВТОМАТИЗИРОВАННАЯ ПРОВЕРКА

### Скрипт для проверки работ студентов

```sql
-- Проверка OLTP
\c ecommerce_oltp;

SELECT 
    CASE 
        WHEN COUNT(*) = 8 THEN 'OK: Все таблицы созданы'
        ELSE 'ERROR: Пропущены таблицы'
    END AS check_result
FROM information_schema.tables 
WHERE table_schema = 'public' AND table_type = 'BASE TABLE';

-- Проверка количества данных
SELECT 
    'users' AS table_name, 
    CASE WHEN COUNT(*) >= 1000 THEN 'OK' ELSE 'ERROR' END AS status
FROM users
UNION ALL
SELECT 'products', 
    CASE WHEN COUNT(*) >= 500 THEN 'OK' ELSE 'ERROR' END 
FROM products
UNION ALL
SELECT 'orders', 
    CASE WHEN COUNT(*) >= 10000 THEN 'OK' ELSE 'ERROR' END 
FROM orders;

-- Проверка OLAP
\c ecommerce_olap;

SELECT 
    CASE 
        WHEN COUNT(*) >= 4 THEN 'OK: Измерения созданы'
        ELSE 'ERROR: Пропущены измерения'
    END AS check_result
FROM information_schema.tables 
WHERE table_schema = 'public' AND table_type = 'BASE TABLE';
```

## 9. ПРИМЕРЫ ВЫПОЛНЕНИЯ

### Пример правильного анализа плана

**Запрос:**
```sql
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'user500@example.com';
```

**Ожидаемый план:**
```
Index Scan using idx_users_email on users  (cost=0.29..8.31 rows=1 width=86)
  Index Cond: ((email)::text = 'user500@example.com'::text)
  Actual Time=0.015..0.016 rows=1 loops=1
  Buffers: shared hit=2
Planning Time: 0.182 ms
Execution Time: 0.035 ms
```

**Правильный анализ:**
> "Используется Index Scan по индексу idx_users_email. 
> Время выполнения 0.035 мс - отличный результат.
> Обработана 1 строка. Cost 0.29..8.31 показывает эффективное использование индекса."

### Пример неправильного анализа

> "Запрос быстрый, всё хорошо."

**Оценка:** 0/5 баллов (слишком поверхностно)
