# КИМ-2.1. Коллоквиум по теме «Подготовка данных»

[Коллоквиум](appendix-01-colloquium_M2.pdf).

## Назначение оценочного средства
Коллоквиум по модулю 2 «Подготовка данных» предназначен для текущего контроля студентов и оценки уровня сформированности компетенций в области интеграции, подготовки и управления качеством данных.
Результаты коллоквиума позволяют преподавателю:
•	Выявить пробелы в теоретических знаниях студентов.
•	Скорректировать содержание дальнейшего обучения.
•	Оценить готовность студентов к выполнению практических задач по подготовке данных.

## Формулировка задания и исходные материалы
Список вопросов по коллоквиуму

**Раздел 1. Консолидация и интеграция данных**
1.	Понятие консолидации данных. Обобщенная схема процесса консолидации. Цели и задачи консолидации данных в корпоративных системах.
2.	Интеграция данных из разнородных источников. Проблемы, возникающие при интеграции. Подходы к решению проблем несовместимости форматов и схем данных.
3.	ETL-процессы: определение, этапы, назначение. Детальное описание каждого этапа: Extract (извлечение), Transform (преобразование), Load (загрузка). Особенности реализации на практике.
4.	ELT-процессы: определение, отличие от ETL. Сравнительный анализ ETL и ELT. Когда какой подход следует применять? Преимущества и ограничения каждого подхода.
5.	Типы данных: структурированные, полуструктурированные, неструктурированные. Примеры каждого типа. Особенности работы с каждым типом данных при подготовке к загрузке в хранилище.
6.	Форматы представления данных. CSV, JSON, XML, Parquet. Сравнение форматов, сценарии использования, преимущества и недостатки.
7.	Инструменты ETL/ELT. Обзор современных инструментов: Apache Airflow, Apache NiFi, Talend, AWS Glue. Назначение, ключевые возможности, сравнение.

**Раздел 2. Подготовка данных к загрузке**

9.	Этапы подготовки данных. Очистка, нормализация, преобразование, обогащение. Детальное описание каждого этапа.
10.	Методы очистки данных (Data Cleaning). Обработка пропусков (удаление, заполнение средним/медианой, интерполяция). Удаление дубликатов. Обработка выбросов. Обработка некорректных форматов.
11.	Нормализация и стандартизация данных. Min-Max нормализация, Z- стандартизация. Когда какой метод применяется? Влияние на качество последующего анализа.

**Раздел 3. Качество данных (Data Quality)**

13.	Понятие качества данных. Определение. Почему качество данных критично для систем ИИ и аналитики?
14.	Основные показатели качества данных (DQ). Полнота (Completeness), точность (Accuracy), своевременсть (Timeliness), согласованность (Consistency), уникальность (Uniqueness), допустимость (Validity). Определение каждого показателя, примеры проверок.
15.	Профилирование данных (Data Profiling). Что это такое? Какие задачи решает? Какие метрики вычисляются при профилировании?
16.	Методы и инструменты контроля качества данных. Apache NiFi, Great Expectations, OpenMetadata. Назначение, ключевые возможности. Как организовать автоматические DQ-проверки в пайплайне?

**Раздел 4. Оркестрация конвейеров данных**

17.	Apache Airflow: назначение и архитектура. DAG (Directed Acyclic Graph), операторы, сенсоры, XCom. Основные понятия и принципы работы.
18.	Организация оркестрации ETL/ELT-процессов с помощью Airflow. Создание DAG, настройка расписания, обработка ошибок, логирование.
19.	Понятие Data Lineage (происхождения данных). Что это такое? Зачем нужно отслеживать происхождение данных? Как это помогает в управлении качеством и доверии к данным?
20.	Инструменты для Data Lineage. OpenMetadata, Apache Atlas. Назначение, ключевые возможности. Как организовать сбор метаданных и построение графа происхождения данных?

**Раздел 5. Обогащение данных**

21.	Понятие обогащения данных (Data Enrichment). Что это такое? Какие задачи решает?
22.	Внутреннее обогащение. Генерация признаков (Feature Engineering). Примеры создания новых признаков на основе существующих.
23.	Внешнее обогащение. Использование открытых API, парсинг веб-данных (web scraping), интеграция с внешними базами знаний. Примеры обогащения данных из внешних источников.

## Правила использования внешних источников, программных инструментов и генеративного ИИ

На коллоквиуме разрешены только минимально необходимые источники. Для подготовки к коллоквиуму разрешается использовать генеративный ИИ только для уточнения сложных понятий, не для генерации готовых ответов.

## Требования к воспроизводимости и самостоятельности выполнения работы

Ответ должен демонстрировать собственное понимание материала. Основные определения должны быть сформулированы точно. Ответ должен быть логически структурирован. Студент должен понимать, как знания применяются на практике.
